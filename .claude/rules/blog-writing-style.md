# ブログ記事の文体ルール

このルールはブログ記事（`src/content/blog/`配下のMDXファイル）を作成・編集する際に適用する。

## 参照ファイル（文体のお手本）

記事を作成する前に、以下のファイルを読んで文体・構成を把握すること：

| カテゴリ           | パス                                                     | 特徴                                               |
| :----------------- | :------------------------------------------------------- | :------------------------------------------------- |
| 個人開発ストーリー | `src/content/blog/2025/dailydose-pwa-development.mdx`    | 課題→解決の流れ、技術スタックの紹介、コスト公開    |
| 技術記事           | `src/content/blog/2025/zod-to-valibot-migration.mdx`     | 表形式での比較、コード例、ステップバイステップ解説 |
| 技術スタック紹介   | `src/content/blog/2025/modern-edge-tech-stack.mdx`       | アーキテクチャ図、表による比較、実装の選定理由     |
| ガジェットレビュー | `src/content/blog/2024/review-monitors-u4025qw-dell.mdx` | スペック表、使用感、メリット/デメリット            |

**重要**: ルールを暗記するより、実際の記事を読んで文体を吸収すること。

## コンテンツマーケティングスキルの活用

ブログ記事を作成・編集する際は、`content-marketing:content-marketer` スキルを活用すること。このスキルは以下の点で記事の品質向上に役立つ：

- SEO最適化されたタイトル・見出し構成
- 読者のエンゲージメントを高める構成
- 適切なキーワード配置
- コンバージョンを意識した記事設計

スキルの呼び出し方法：

```text
Skill: content-marketing:content-marketer
```

## 基本トーン

- **丁寧語（です・ます調）** を一貫して使用する
- 一人称は「僕」を基本とし、カジュアルな文脈では「俺」も可
- 直接的で情報豊富なスタイルを心がける
- 冗長な表現を避け、簡潔に要点を伝える

## 記事構成

### 導入部

- 簡潔な挨拶または直接本題へ入る
- 例: 「タイトルにあるモニターを購入しました。」
- 旅行記等では: 「明けましておめでとうございます。フナイです。」

### 本文

- H2/H3で明確に構造化する
- 仕様や比較にはMarkdown表を使用する
- リストは箇条書きで整理する
- 引用は `>` で明示し、出典を記載する

### 結論部

- レビュー記事は「総評」セクションで締める
- 後日の更新は「追記」セクションに日付付きで記載
- カジュアルな締め: 「それでは、またね。」「よろしくお願いします。」

## 文章表現

### よく使う表現

- 「非常に〜」（強調時）
- 「〜ことが可能です」「〜することができます」
- 「〜と思います」「〜と考えています」（個人的意見）
- 「一方で」「具体的には」「当然ながら」（接続・展開）
- 「〜ので」「〜ため」（理由）
- 「普通に〜」（副詞的に。「普通に良いです」「普通に最悪でした」など）
- 「本当に〜」「めちゃくちゃ〜」（素直な強調）
- 「〜すぎます」「〜すぎました」（誇張表現。体験の強さを伝える）
- 「〜かなと思います」（自問・迷いを読者と共有する）

### 括弧内補足（ツッコミ・補足）

筆者の最も特徴的な文体。本文に括弧でツッコミや補足情報を挿入する二層構造:

- 例: 「ちょっと緊急で回しています（洗濯機です）」
- 例: 「ICL（半額キャンペーン）とドラム式洗濯機の出費が重なりました」
- 例: 「レビュー速度より開発速度が速いので一生審査が終わりません（再サブミットしました）」

括弧内はカジュアル寄りにして、本文との温度差でユーモアを生む。

### 自虐ユーモア

積極的に取り入れる。ドライに・淡々と自分の失敗や不幸を語るスタイル:

- 失敗談を大げさにせず事実として報告する
- ~~取り消し線~~で本音やツッコミを挿入する
  - 例: 「~もちろん黒歴史になる可能性も非常にある。~」
- 「〜なのでした」「〜のであった」のような物語調の締めも効果的
- 自分の体験を客観視して面白がるトーンを意識する

### 強調の使い方

- **太字**: 重要なポイントや結論を強調
  - 例: 「**デスクの大きさは心のゆとりに繋がること** を経験から知っていた」

## カテゴリ別の書き方

### ガジェットレビュー（gadgets）

- 製品仕様を表形式で記載
- 以前の製品との比較を含める
- 実際の使用感と具体的なユースケース
- 価格と価値の評価
- 設定やカスタマイズのTips

### 技術記事（programming）

- コード例はシンタックスハイライト付きで記載
- 外部ドキュメントへのリンクを適宜挿入
- なぜその技術が重要かを説明（SEOメリット等）
- 実装手順をステップバイステップで解説

### 旅行記（travel）

- 日ごと（Day 1, Day 2...）の構成
- 行程表を表形式で記載
- 食事・宿泊先の情報
- 個人的なエピソードや感想
- 写真を多用し、説明キャプション付き

### 体験記（lifestyle）

- 体験の時系列に沿った構成（きっかけ→準備→当日→結果）
- 具体的な数値やコストを包み隠さず記載
- 自虐ユーモアや括弧内補足を活かし、読みやすさを確保
- 不安・緊張などの感情も率直に書く
- 参考: `src/content/blog/2026/icl-surgery-experience.mdx`

### 車・乗り物（vehicles）

- 市場分析と背景情報
- スペック比較表
- 個人的な見解と検討事項
- 参考リンクを複数記載

## フォーマット規則

### リンク

- 裸URLは自動的にリンクカードになるため、単独行に配置
- インラインリンクは `[説明テキスト](URL)` 形式

### 画像

- alt属性に日本語で説明を記載
- 画像の直後に補足説明を入れる

### 表

- ヘッダーは日本語/英語混在可
- 左揃え `:---` 、右揃え `---:` を適切に使用

### コード

- タイトル付きシンタックスハイライトを使用
- コードの目的を事前に説明

## 避けるべきこと

- 過度に形式的・堅苦しい表現
- 曖昧な表現（「たぶん」「かもしれない」の多用）
- 根拠のない主張
- メリットのみ・デメリットのみの偏った評価
